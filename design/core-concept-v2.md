# notifyd Core Concept

## 背景

CLI 実行、AI エージェントの応答完了、hook 実行結果など、ローカル開発中には「完了した」「失敗した」「確認が必要」といったイベントが頻繁に発生する。
しかし現状は、それぞれが個別の通知手段に閉じており、tmux・Linux デスクトップ通知・Termux などへの一貫した配信基盤がない。

そのため、

- 長時間コマンドの終了に気づけない
- 別 pane / 別 workspace で走らせた処理の失敗を見落とす
- ツールごとに通知連携を個別実装する必要がある
- 通知先の追加や差し替えのたびに実装が散らばる

という問題がある。

## 目的

ローカル環境で発生するイベントを一箇所に集約し、複数の通知先へ柔軟に配送できる軽量な通知ハブを作る。

この基盤により、以下を実現する。

- CLI / hook / SDK から統一的にイベント送信できる
- `notify subscribe` で SSE 経由のリアルタイム受信 + ローカルコマンド実行
- sender / subscriber を疎結合にできる
- 今後通知先やイベント送信元が増えても拡張しやすい

## 前提: なぜ command はクライアント側で動くべきか

`notify-send` や `tmux display-message` は、**送信元プロセスのコンテキスト**で実行されるべき。

- デスクトップ通知は送信元の `DISPLAY` / `DBUS_SESSION_BUS_ADDRESS` で表示される必要がある
- tmuxコマンドは送信元のtmuxセッションで実行される必要がある
- daemonプロセスにはこれらの環境変数が無い可能性が高い

つまり、**command 実行は「イベントを購読したプロセスが、自分のコンテキストで実行する」**のが正しいモデル。

## アーキテクチャ

```
┌─────────────┐
│ CLI (send)   │
└──────┬──────┘
       │
       └────────────────┘
                        │
                 ┌──────┴──────┐
                 │   notifyd   │
                 │  (daemon)   │
                 └──────┬──────┘
                        │ SSE broadcast (全クライアントに配信)
           ┌────────────┼────────────┐
           │            │            │
    ┌──────┴──────┐ ┌───┴────┐ ┌────┴─────────┐
    │  subscribe  │ │  sub   │ │  subscribe   │
    │  --command  │ │        │ │  --command   │
    │  echo hi   │ │  ...   │ │  notify-send │
    └─────────────┘ └────────┘ └──────────────┘
```

- **notifyd** はイベントを受け取り、SSE で全接続クライアントにブロードキャストする
- **notify subscribe** は SSE でイベントを受信し、ローカルでコマンドを実行する
- クライアント側で `--group-filter` によるフィルタリングが可能

### send の流れ

### subscribe の流れ

```
CLI (subscribe)                    daemon
  │                                  │
  │  GET /events/stream              │
  │ ───────────────────────────────> │
  │ <─────────────────────────────── │  ← SSE接続確立
  │                                  │
  │  SSE: event データ               │  ← 他のCLIからのイベントが届く

  │                                  │
  │  ローカルコマンドを実行          │
  │  (notify-send, tmux等)          │
  │                                  │
```

## 設計上の決定

### daemon はイベントをブロードキャストするのみ

daemon はイベントを受け取り、SSE で全接続中のクライアントに配信するだけ。コマンドの実行は行わない。

### クライアント側でコマンドを実行する

`notify subscribe --command '...'` が SSE でイベントを受信し、ローカルでコマンドを実行する。これにより `DISPLAY` や tmux セッションなど送信元のコンテキストが利用可能。

### クライアント側でフィルタリングする

`--group-filter` オプションで特定のグループのみ受信可能。daemon は全イベントをブロードキャストし、フィルタリングは各クライアントが担当する。

### コマンドテンプレート変数

クライアント側で実行されるため、送信元プロセスの環境変数がそのまま使える:

| 変数 | 内容 |
|------|------|
| `${title}` | タイトル |
| `${message}` | メッセージ |
| `${level}` | レベル |
| `${group}` | グループ名 |

環境変数 `NOTIFYD_EVENT` に JSON 形式でイベント全体が渡される。

### 永続化しない

group 情報はメモリ上のみ。再起動でリセットされる。必要になった段階でのみ永続化を検討する。

### 配信保証は弱い

SSE 接続が切れた場合のイベントは消失する。通知用途であり、まずは過剰に重くせず、必要になった時点で再接続やキューを追加する。

### 配送ハブに徹する

notifyd はイベントの受信・ブロードキャストのみ。履歴管理や再送は扱わない。

## データモデル

### Event

- id — サーバー生成
- time — サーバー生成
- title — イベントタイトル
- message — 省略可
- level — info | warn | error (default: info)
- group — 省略可。指定時はスキーマバリデーションが走る
- process_id — 省略可
- tags — 省略可
- meta — 省略可。group 指定時はスキーマでバリデート

### Group

- name — グループ名
- schema — JSON Schema。meta の構造を定義
- description — 省略可
- created_at — サーバー生成

※ いずれも daemon 再起動時には失われる前提とする。

## 非スコープ

初期段階では以下は後回しにする。

- イベント永続化
- 配信失敗時の再送
- backlog / replay
- 厳密な queue 制御
- Web UI
- 複数端末間同期
- 高度な認証認可
